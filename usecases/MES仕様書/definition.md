# ユースケース: MES 仕様書

ENAA「MES/MOM 導入のための標準業務一覧」を骨格として、自社の MES 仕様書 / RFP 一次稿を対話形式で組み立てる。

## 出典・前提（重要）

本ドキュメントが規定する対話フロー・出力スキーマ・分類体系は、すべて以下の ENAA 資料に基づく：

- **G25-029-1**: MES/MOM 導入のための標準業務一覧 解説資料
- **G25-029-2**: MES/MOM 導入のための標準業務一覧（Excel 本体）
- **G25-029-3**: クイックスタートガイド
- **G25-029-5**: How to Use

| 項目 | 内容 |
|---|---|
| 公開元 | 一般財団法人エンジニアリング協会(ENAA) 次世代スマート工場のエンジニアリング研究会 |
| 公式サイト | https://www.enaa.or.jp/research/smart/mes |
| バージョン | 2025 年 9 月正式版 |
| 利用条件 | 自社内・RFP への二次利用可、**著作権表示必須** |

本雛形は ENAA の構造・方法論を参照する **orchestrator** であり、ENAA 資料の置き換えを意図しない。具体的な業務分類の詳細・業務オペレーション記述・代替可能システム情報は、利用者が ENAA 公式サイトから資料をダウンロードして `reference/ENAA/` に配置することで初めて利用可能になる。

本ドキュメント中に登場する具体的な Division 名・Business Category 名は、利用者が ENAA 公式サイトに辿り着くための **道案内** として記載している。

## 対象ユーザー

製造IT 担当者 / SIer / MES ベンダーのコンサル / 工場 DX 推進責任者

## 対象スコープ（ENAA の位置付けに準拠）

ENAA Quick Start（G25-029-3 p10）の規定により、本雛形は **「企画・要件定義」フェーズの中の「システム化計画」** を対象とする。

```
[企画・要件定義]                      [開発] [運用] [保守]
 システム化の方向性 → システム化計画 → 要件定義 → 設計 → 実装 → ...
                       ★ ここが対象 ★
```

「システム化の方向性」（MES/MOM の価値を理解するか／導入すべきか）は、別途 ENAA が出版企画中の書籍等でカバー予定とされており、本雛形の主目的ではない。
「要件定義」以降は本雛形の出力（RFP 一次稿）をベンダー・パートナーに渡す段階。

## 入力

- ENAA 5 資料（`reference/ENAA/`、利用者が事前配置）
- ENAA 12 大分類リファレンス（`usecases/MES仕様書/ENAA-業務分類体系.md`）
- 利用者からの対話入力（4 段階の処理フロー参照）

## 4 段階の処理フロー（ENAA Quick Start 推奨手順）

ENAA Quick Start（G25-029-3 p7-15）が「システム化計画」フェーズで推奨するアクティビティを 4 段階に整理。
各段階で対話を進め、最後に YAML / Markdown を生成する。

### Stage 0: 前提確認・案件メタ聴取

1. **ENAA 配置確認＋バージョン取得** → `extraction-recipes.md` の **R0** を実行
   - 不備があれば `core/rules.md` のフォールバックフローへ
2. **全構造の動的把握** → **R1** を実行して Division × BC × MES 対象比率を取得
   - 開発時は `usecases/MES仕様書/ENAA-業務分類体系.md` の静的リファレンスでも代替可（private 限定）
3. **既存カスタム書き込みの確認** → **R5** を実行
   - 利用者が ENAA Excel の P 列以降に既に書き込んでいる場合は引き継ぎを提案
4. **案件メタを聴取**（1問1答、5〜7 問）
   - 案件名 / 対象工場（discrete 想定）/ 期間 / 予算感 / 適用範囲 / 既存システム概要 / 担当部門

### Stage 1: 調査星取表（ENAA Quick Start p11 推奨）

> 「自社の製造領域システムの調査と課題の洗い出し」段階。
> ENAA 標準業務一覧で **星取表形式** に既存システムのカバー範囲を洗い出す。

対話手順：

1. 12 大分類を **MES 対象比率の高い順**（中核 5 → 中間 2 → 周辺 5）で順に提示し、各 Division について 4 値で確認：
   - **必須** / **任意** / **不要** / **未定**
   - （優先度判定は Stage 0 の **R1** 結果から `mes_on/total` 降順）
2. 「必須」「任意」と回答された Division について、**R2** を実行して BC × Sub BC 階層を取得 → Business Category 単位で深掘り：
   - 該当業務あり（○ / ×）
   - 現業務媒体（既存システム名 / Excel / 手書き / その他）
   - 現状業務の概要
   - 現状の課題（複数可）
3. 利用者が「特定 Sub BC の中身が見たい」と言ったら **R3** を実行して該当 Business Process（業務オペレーション・補足込み）を提示
4. 「不要」「未定」は理由を 1 文だけ記録して次へ
5. 中核 5 分類はスキップ不可（少なくとも 4 値の判定までは必ず取る）。周辺 7 分類は「該当なし」が大半なら一括で「不要」確認可

→ Stage 1 出力：`調査星取表` セクションの YAML

### Stage 2: 自社業務 / システム課題の整理（ENAA Quick Start p12-13 推奨）

> Stage 1 で挙がった課題と未カバー領域を「要求機能一覧」へ落とし込む段階。

対話手順：

1. Stage 1 の課題を一覧化して提示
2. 各課題に対する **改善・実現したいこと（To-Be 施策）** を聴取
3. 各施策を「機能」として要求機能一覧に登録（複数の課題が同一機能に紐づくこともある）
4. 利用者が「この機能は既存の何で代替できる？」と聞いてきたら **R4**（特定 BP_ID の代替可能システム）を実行して ENAA の Integration Sample を提示
5. **優先順位付け**:
   - 効果の大きさ / 早さ / 難易度 をベースに 高 / 中 / 低 を決定
   - 実現時期（年度・半期レベル）を仮置き
   - As-is / To-be の対比を要件ごとに記述
6. ENAA 推奨の追加項目（Excel P 列以降に対応）を網羅:
   - 担当部門 / 該当業務有無 / 現業務媒体 / 現状 / 課題 / To-Be / 施策

→ Stage 2 出力：`要求機能一覧` セクションの YAML

### Stage 3: 期待効果と ROI 算出（ENAA Quick Start p13-14 推奨）

> 「効果・ROI 算出、以降のフェーズの体制・予算の確保（概算レベル）」段階。

ENAA が示す **MES/MOM の 3 つの効果軸** で構造化する：

| 効果軸 | 内容 | 定量化 |
|---|---|:---:|
| **1. 生産性・品質の向上** | 効率性 / 正確性の向上、定型業務からの解放 | しやすい |
| **2. 新しい能力の獲得** | 自動計算・即時対応、新たな業務の創出 | しやすい |
| **3. 判断の質の向上** | データに基づくサジェスト、正しい判定 | しにくいが**重要** |

対話手順：

1. 課題ごとに 3 効果軸のいずれに該当するかを判定（複数可）
2. 定量効果は現状値・目標値・金額換算を聴取
3. 定性効果（特に「判断の質の向上」）は具体例で聴取
   - ENAA はカーナビのメタファーを推奨：「現在地・残距離・到着予測を知ることで判断が変わる」と同様の効果を MES に当てはめる
4. ROI 概算：投資見積 vs 改善金額累積 + リスク回避額
5. 投資見積が出ていない場合、目標 ROI 年数からの逆算で**投資予算ターゲット値**を提示

→ Stage 3 出力：`期待効果` および `ROI概算` セクションの YAML

### Stage 4: ロードマップと RFP 一次稿生成（ENAA Quick Start p12, p15 推奨）

> 「中長期ロードマップ作成」「パートナーへの提案依頼（RFP 発行）」段階。

対話手順：

1. ロードマップ：Phase1 / Phase2 等で実現時期と範囲を確定
2. 非機能要件（パフォーマンス / 可用性 / セキュリティ / 既存基幹系連携）を聴取
3. RFP 想定（発行時期 / 候補パートナー / 選定基準）を聴取
4. 全段階のデータを統合して以下 2 ファイルを保存：
   - `outputs/MES仕様書/<YYYYMMDD>_<案件名>.yaml`（構造化データ）
   - `outputs/MES仕様書/<YYYYMMDD>_<案件名>.md`（RFP 一次稿）
5. **ENAA 出典記載が両ファイルに含まれているか検証**（`core/rules.md` 強制仕様）
6. ファイルパスを利用者に伝え、`/保存` 案内

## 出力 Markdown 章立て（RFP 一次稿）

ENAA Quick Start p13 が示す「実行カテゴリー（Division ÷ Business Category 程度の粒度）」と「実行タスク（Sub Business Category 程度の粒度）」の構造を踏襲。

1. **案件概要**（社名・対象工場・期間・予算感・担当部門）
2. **位置付け**：ENAA 推奨の「企画・構想 → システム化計画」フェーズの一次稿である旨を明記
3. **調査星取表**：12 大分類 × 自社既存システムカバー状況（Stage 1 結果）
4. **要求機能一覧**：機能 ID / 分類 / As-Is / To-Be / 施策 / 優先度 / 実現時期（Stage 2 結果）
5. **期待効果**：MES/MOM 3 効果軸（生産性品質 / 新能力 / 判断の質）（Stage 3 結果）
6. **ROI 概算**：投資・改善・回避額・ROI 年数（Stage 3 結果）
7. **中長期ロードマップ**：Phase 1 / Phase 2 ...（Stage 4 結果）
8. **非機能要件**：パフォーマンス / 可用性 / セキュリティ / 既存基幹系連携
9. **RFP 発行に向けて**：発行時期 / 候補パートナー / 選定基準
10. **制約・前提**
11. **出典・参考**：ENAA G25-029-1 / G25-029-2（必須記載）

## 用語確認・ISA-95 マッピング等の補助

利用者が「ISA-95 のレベル 2 ってどういう意味？」「ENAA は MOM をどう定義している？」「MBOM と EBOM の違いは？」等の用語確認を求めてきた場合：

- 開発時（private）: `usecases/MES仕様書/ENAA-用語集.md` を即時参照（38 用語が Division 別に整理済み）
- 利用時（public）: `extraction-recipes.md` の **R6**（PDF キーワード検索）を解説資料 (G25-029-1) または How to Use (G25-029-5) に対して実行して、ENAA 自身の定義から引用

ISA-95 関連の用語（L2 / SCADA / DCS）は `ENAA-用語集.md` の「ISA-95 マッピング」セクションに整理済み。深掘りには IEC 62264 / ISA-95 規格本体を別途参照する旨を案内する。

## 残課題（v0.2 以降）

- **メモリキャッシュ**: 現状 v0.1.5 ではレシピがステートレスで毎回 Excel を直読。Stage 0 で R1 を一度実行してセッション中保持する仕組みに進化させると 2 回目以降のクエリが高速化（`extraction-recipes.md` 末尾の v0.2 検討事項を参照）
- **業務機能関連図のテキスト化**: G25-029-1 解説資料 P7-11 の図表は pypdf では拾えないため、別アプローチ（OCR 等）を検討
- **ISA-95 / IEC 62264 対応マトリクス**: R6 で都度参照する以上の体系的なマッピング表が必要なら別途整備
- **業務機能関連図のテキスト化**: ENAA の図表（解説資料 P7-11）を Claude が読み取れるテキスト形式で別途整備
- **架空企業ベースのサンプル outputs**: 中堅食品卸（business-os 内サンプルと同種）でなく、ディスクリート系（自動車部品メーカー想定など）のサンプルを 1 件作成
- **MES/MOM の 3 つの効果のテンプレート例文集**: 「判断の質向上」効果は記述難易度が高いので、業種別の例文集を整備
- **ENAA RFP 作成テンプレート**との整合（ENAA 別途取得要、未着手）
- **プロセス系・バッチ系工場用の分岐**: 現状 ENAA は discrete 限定。process 業界では別の業界標準（NAMUR等）への翻訳が必要
