# セキュリティと情報漏洩対策

> 製造業の MES/MOM 検討では、競合に流出すれば致命的な情報（顧客名・取引条件・ライン構成・歩留まり数値・原価情報など）を扱います。
> Claude Code（AI 対話ツール）を使う以上、**この前提を理解した上で適切な運用を選ぶ**必要があります。
>
> 本ドキュメントは利用開始前に必ず一読し、可能であれば**情シス・法務担当者と一緒に確認**してください。

## 1. まず理解しておくべき：Claude Code のデータフロー

Claude Code は **クラウド型の AI サービス** です。完全ローカルで動くツールではありません。

```
┌──────────────┐     インターネット (HTTPS)    ┌──────────────────┐
│ 利用者の PC  │ ─────────────────────────► │ Anthropic の API │
│              │ ◄───────────────────────── │  （米国）        │
└──────────────┘                             └──────────────────┘
   ↓ ローカル                                          ↑
   reference/ENAA/                                    送信される情報:
   inputs/                                            - 利用者の入力テキスト
   outputs/                                           - Claude が「読んだ」ファイルの中身
                                                       - CLAUDE.md / core/rules.md 等の指示書

→ Claude の応答が返ってきて画面に表示
```

### 何が送信されるか / されないか

| 操作 | Anthropic に送信される？ |
|---|:---:|
| 利用者が入力した質問・指示文 | **送信される** |
| `reference/ENAA/*.xlsx` を Claude が「読んで」と指示された | **送信される（中身全体）** |
| `inputs/<機密ファイル>` を Claude が読んだ | **送信される（中身全体）** |
| `outputs/` に Claude が書き出したファイル | **送信される（書き出した内容そのもの）** |
| `outputs/.cache/enaa_full.json` の存在 | キャッシュ自体は送信されないが、Claude が参照すれば中身は送信される |
| ファイル名の言及（中身を読まない） | 名前だけ送信、中身は送信されない |
| Claude を起動しているだけ | 起動だけでは送信されない（対話開始で送信が始まる） |

> **重要な事実**：「ローカルに置いた」だけでは情報は外部に出ません。**「Claude にそのファイルを処理させる」と決めた瞬間に、その内容が Anthropic API に送信されます**。

## 2. 利用前に必ず確認すべきこと

### 2.1 Anthropic のプラン・データ利用ポリシー

利用するプランによって、Anthropic がデータをどう扱うかが異なります（**契約時に必ず最新のポリシーを公式サイトで確認**）：

| プラン | データ学習に使われる？ | 推奨度（製造業機密用途） |
|---|:---:|:---:|
| Claude Free | （利用者が明示同意した場合のみ。要確認） | ❌ 不可 |
| Claude Pro（個人サブスク） | デフォルト学習に使われない（公式声明） | △ 個人検証用途まで |
| Claude Team | 学習に使われない、企業向け管理機能あり | ○ 小規模法人向け |
| Claude Enterprise | 強化されたデータ分離、ログ管理 | ◎ **推奨** |
| API 直接利用 | デフォルト学習に使われない | ○ 高度な制御が必要な場合 |

**最低条件：Claude Pro 以上**を強く推奨。Free プランでの機密データ取扱いは避けてください。

> 公式ポリシーの URL は変わることがあるため、必ず Anthropic 公式サイト（claude.ai / anthropic.com）の最新版を確認してください。本ドキュメントの記述と公式ポリシーに齟齬があれば、**公式ポリシーが優先**です。

### 2.2 社内ルール・契約上の許諾

導入前に **必ず情シス・法務に確認**：

- [ ] AI ツールの業務利用が社内ルール上許可されているか
- [ ] 顧客データを外部 SaaS に渡すことが取引契約（NDA / 個人情報保護条項）に抵触しないか
- [ ] トヨタ系・自動車業界等の **取引先からのセキュリティ要件**（IATF 16949 / TISAX 等）に抵触しないか
- [ ] ISMS / プライバシーマーク等の認証要件に違反しないか
- [ ] 米国経由のサービス利用（Anthropic は米国企業）が経済安全保障・輸出管理上問題ないか

### 2.3 ネットワーク・端末

- [ ] 業務ネットワーク経由で利用する（自宅 PC / 公共 Wi-Fi 不可）
- [ ] VPN・プロキシの使用ルールに従う
- [ ] 業務貸与端末で利用（個人端末での社用機密処理は禁止が一般的）
- [ ] エンドポイント保護（EDR / DLP）が稼働している

## 3. 機密データの取扱い指針

### 3.1 「Claude に渡す前に伏せ字化」を徹底する

本スターターは、機密性が疑われる入力に対して伏せ字化を提案する設計です（`core/rules.md`）。
ただし**最終判断は利用者**です。以下のような情報は、必ず Claude に送る前に伏せ字化してください：

| 種別 | 伏せ字化の例 |
|---|---|
| 顧客名 | A 社 / B 社 / 顧客 X |
| 取引先名 | サプライヤー A / 仕入先 B |
| 製品名・型番 | 製品 A / 型番 X-001 |
| 部品コード | コード A1 / 部品 1 |
| 担当者氏名 | A 担当 / Z 課長 |
| 工場の住所詳細 | 関東某拠点 / 関西工場 |
| 売上金額・原価の絶対値 | 月数千万円〜億円規模 |
| ライン番号・設備固有 ID | ライン A / 設備 1 |

### 3.2 「文脈は残す、固有名詞だけ消す」

伏せ字化のコツは **会話の意味は失わず、識別可能な情報だけ消す** こと。

❌ 過剰伏せ字（意味が壊れる）：

```
A について B と一緒に C を作りたい
```

✅ 適切な伏せ字（意味は通る）：

```
当社の主力製品（製品 A）について、サプライヤー A と一緒に新しい工程を作りたい
```

### 3.3 数値の扱い

- **絶対値は出さない**：「月商 3.2 億円」→「数億円規模」
- **比率や相対値は OK**：「不良率 0.3%」「目標 0.15%」
- **時間・期間は OK**：「2027-04 稼働」
- **ROI・効果額の概算**：丸めて記述（「約 9000 万円/年」程度）

### 3.4 inputs/ にファイルを置くときの判断基準

`inputs/` フォルダに自社業務リスト等を置くことは、`.gitignore` で git からは除外されています。
ただし **Claude にそのファイルを読ませた瞬間、内容が Anthropic に送信されます**。

判断のフロー：

```
1. このファイルを Claude に処理させる必要があるか？
   ├ いいえ → inputs/ に置かない（手元のファイルから手動で要点だけ Claude に伝える）
   └ はい → 次の質問へ

2. このファイルに伏せ字化されていない機密情報があるか？
   ├ ある → まず伏せ字化したコピーを inputs/ に置く（原本は別フォルダで保管）
   └ ない → そのまま Claude に渡してよい

3. Claude が処理した結果、outputs/ に機密が混入する可能性は？
   ├ ある → outputs/<案件>/ ごと .gitignore に追加
   └ ない → デフォルトの追跡で OK
```

## 4. inputs / outputs / .cache の扱い

### 4.1 `inputs/`（利用者の機密データ置き場）

- `.gitignore` で git から除外済み
- ただし **ローカルファイルとして残るので、PC 紛失・盗難リスクは別途対策**
- BitLocker / FileVault 等のディスク暗号化を有効化推奨

### 4.2 `outputs/`（生成された仕様書）

- デフォルトで git 追跡対象
- **顧客名・案件名がファイル名に入る**ことが多いので注意
- 機密案件は **案件単位で `.gitignore` に追加** する：

```gitignore
# .gitignore に追加する例
outputs/MES仕様書/<機密案件名>/
```

- public リポへの push は絶対にしない（mes-spec-os は **ローカル運用前提** の設計）

### 4.3 `outputs/.cache/`（自動生成キャッシュ）

- ENAA 公開データの構造化キャッシュ
- 機密性は低い（ただし将来的に Excel の P 列以降に利用者カスタムが入ると機密化）
- `.gitignore` で除外済み

## 5. 案件開始時のチェックリスト

新しい MES 案件で本スターターを使う前に：

- [ ] 取り扱う情報の機密区分を社内基準で確認した
- [ ] 顧客との NDA や取引契約に AI ツール利用の制約がないか確認した
- [ ] 使用する Claude プランがデータ学習に使われない契約か確認した
- [ ] 端末・ネットワークが社内セキュリティ要件を満たしている
- [ ] `inputs/` に置く予定のファイルを伏せ字化済み or 伏せ字化計画ありか
- [ ] `outputs/` の追跡方針を決めた（git 追跡 / ignore / 別保管）
- [ ] チームメンバー全員が本ドキュメントを読んでいる

## 6. 案件終了時のチェックリスト

- [ ] `outputs/` のファイルを社内ファイルサーバーへ正式保管に移行した
- [ ] `inputs/` の機密ファイルを安全に削除（できれば shred 等で完全消去）
- [ ] git に commit してしまった機密がないか `git log` で確認
- [ ] 機密が混入していた場合：履歴書き換え（`git filter-branch` / BFG Repo-Cleaner）
- [ ] Claude Code の対話履歴（`~/.claude/projects/...`）に残った機密を確認・削除

## 7. リスクシナリオと対応

### シナリオ A: 機密が含まれた状態で Claude に渡してしまった

**起きること**: その時点で Anthropic API に送信済み。取り消し不可。

**対応**:
- 利用しているプランの **データ保持期間** を確認（30 日〜90 日が一般的）
- Anthropic サポートへ削除依頼が可能か確認（Enterprise プランでは申請窓口あり）
- 社内インシデント手順に従って報告
- 同じ機密が漏洩した可能性のある経路（メール・チャット等）も同時に確認

### シナリオ B: outputs/ を public リポに push してしまった

**起きること**: GitHub 等の public リポにアップされ、世界中から閲覧可能になる。

**対応**:
- **時間との勝負**：即座に該当 commit を revert + force push
- ただし **既にクロールされた可能性が高い**ため、漏洩前提で対応
- GitHub の場合：[Sensitive data removal](https://docs.github.com/en/code-security/secret-scanning) ヘルプを参照
- 関係者へ報告、社内インシデント手順に従う
- private 運用が前提のスターターなので、**そもそも push しない設計**を徹底

### シナリオ C: PC 紛失・盗難

**起きること**: ローカルの `inputs/` `outputs/` が物理的に流出。

**対応**:
- ディスク暗号化（BitLocker / FileVault）が有効なら被害は限定的
- 暗号化なしの場合：紛失と同時にすべて漏洩したと想定して対応
- 社内紛失盗難手順に従う

## 8. 「クラウド AI を使えない」場合の代替

社内ルール・契約上 Claude Code が使えない場合：

- **本スターターを直接利用するのは不可**
- 代替案：
  - ENAA 標準業務一覧 Excel を **手動で**埋める（Claude 対話なし）
  - 社内承認済みのオンプレミス LLM（Claude API 経由ではなく自社環境）を立てて、本スターターの設計思想（4 段階フロー / 3 効果軸 / 星取表）を参考にする
  - ENAA に直接コンタクト（`enaa-mes (at) enaa.or.jp`）

本スターターの **設計思想と章立てそのものは AI ツールに依存していない** ため、Excel + Word でも同じ手法が実行可能です。

| ベンダーロック | なし（手法は AI 非依存で再実装可能） |
