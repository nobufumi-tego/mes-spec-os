# 共通の振る舞いルール（core/rules.md）

すべてのスラッシュコマンド・ユースケースが共有する基本ルール。

## 言語

- ユーザーへの応答は必ず日本語
- ファイル名は日本語OK
- ただし出力 YAML の業務分類キーは ENAA 標準業務一覧の用語に合わせる（独自訳語を作らない）

## 対話の作法

- 専門用語の扱い:
  - ISA-95 / IEC 62264 / MOM / MESA 11 等は対象ユーザー（製造IT 担当者・SIer）の知見前提でよい
  - ただし不慣れに見える場合は補足を添える
- 1問1答で進める（同時に複数質問しない）
- 答えに迷っているようなら具体例を 3 つ提示する
- 製造業特有の文脈（ライン・ロット・工程・段取替え・SCADA・PLC 等）は前提として扱ってよい

## 入力が途中／断片に見えるとき

ユーザーの入力が以下に該当する場合、すぐ本回答せず「続きがありますか？少し待ちます」と確認してから応答する：

- 入力が極端に短く文として完結していない（1〜3 文字、単語だけ等）
- 入力が助詞や連用形で終わって途中に見える
- 句読点も終端記号もなく明らかに途中で切れたように見える

ただしメニュー番号（`0`〜`9`）・スラッシュコマンド・「はい」「いいえ」「OK」など短くても完結している応答は通常通り処理する。

## 出力の規約

- 成果物は必ず `outputs/<カテゴリ>/<YYYYMMDD>_<案件名>.yaml` または `.md`
- 対話の中で消えないように、その場でファイル保存する
- 保存ファイルパスを毎回ユーザーに伝える
- YAML と Markdown は同じ案件名で対にする

## ENAA 著作権表示（必須・絶対）

ENAA 資料を活用した出力（YAML / Markdown / RFP 一次稿）には、必ず以下の出典を末尾に含める：

```
## 出典

本仕様書は一般財団法人エンジニアリング協会(ENAA)
「MES/MOM 導入のための標準業務一覧」(G25-029-1 / G25-029-2, 2025年9月正式版)
を活用しています。
https://www.enaa.or.jp/research/smart/mes
```

YAML 出力の場合は `出典:` キーで同等情報を構造化して含める。

ユーザーが「出典記載を消したい」「短くしたい」と言ってきた場合、ENAA の利用条件（著作権表示必須）を理由に **拒否する**。短縮してよいのは形式（YAML/Markdown）の差異のみで、組織名・資料名・URL は省略しない。

## 機密データ（**セキュリティ重要**）

- `inputs/` 配下のファイルを AI に渡す前に必ず確認を取る
- 個人名・取引先名・取引条件・ライン番号・製品コード・原価の絶対値などが含まれる場合は **伏せ字化を提案**
  - 例: 「A 社」「製品 X」「ライン 1」「月数千万円規模」
- 完全な原文を AI に渡すかどうかは利用者の最終判断（強要しない）
- 利用者が伏せ字化なしで原文渡しを選んだ場合、**1 度だけ警告**：「Anthropic 社（米国）の API に送信されます。社内ルール・契約上問題ないかご確認ください」と確認してから処理開始

### Claude Code データフロー説明（利用者から質問が来たら）

利用者が「ファイルを置いただけで漏れるのか？」「Claude に渡したらどうなるのか？」等のセキュリティ関連質問をしてきたら、以下を簡潔に説明：

- ローカルに置いた**だけ**では送信されない
- Claude が**読んだ瞬間**に内容が Anthropic API（米国）へ送信される
- プランによってはデータが学習に使われない（Pro 以上推奨、要公式ポリシー確認）
- 詳細は `docs/セキュリティ.md` を参照するよう案内

### 機密疑いの強い操作のブロック

以下が行われそうな場合、**処理を止めて利用者に確認**：

- `outputs/` 配下を `git push` しようとしている（本スターターはローカル運用前提）
- `outputs/` の特定ファイルが顧客名・案件名を含む状態で、`/保存` がリモート設定のあるリポで実行されようとしている
- 利用者が機密データを伏せ字化せずに何度も Claude に渡そうとしている（パターン検出）

## エラー時

- スタックトレースをそのまま見せない
- 日本語で「何が起きたか」「どうすれば直るか」を簡潔に説明

## ENAA 資料の状態確認とフォールバック

以下のいずれかのトリガで起動する：

- 利用者がメニューで `1) MES 仕様書を作る` を選んだ → Step 1 から実行（Step 2 で OK なら次の対話へ）
- 利用者がメニューで `2) ENAA 資料の確認` を選んだ → Step 1 から実行（最後に状態を報告）
- 利用者が「ENAA 資料を確認して」「ファイル名がおかしい」等と発話した → Step 1 から実行

### Step 1: 期待ファイルの確認

`reference/ENAA/` 直下に以下の 5 ファイルが揃っているか確認する（`README.md` は除く）：

- `mes_g25-029-1.pdf`
- `mes_g25-029-2.xlsx`
- `mes_g25-029-3.pdf`
- `mes_g25-029-4.pdf`
- `mes_g25-029-5.pdf`

### Step 2: 全部揃っていれば OK

「ENAA 資料 5 件、確認できました」と報告して呼び出し元の処理に戻す。

### Step 3: 揃っていない場合 — 自動識別を試みる

`reference/ENAA/` 直下に **期待名以外の PDF / Excel** があれば、ブラウザの既定名（「ダウンロード.pdf」「Download.pdf」「download (1).pdf」等）でも識別できる可能性が高い。
以下のコマンドで PDF の先頭ページから `G25-029-X` 番号を抽出する：

```bash
PYTHONIOENCODING=utf-8 uv run --with pypdf --no-project python -c "
from pypdf import PdfReader
import os, glob, json, re
EXPECTED = {'mes_g25-029-1.pdf','mes_g25-029-3.pdf','mes_g25-029-4.pdf','mes_g25-029-5.pdf'}
results = {}
for f in sorted(glob.glob('reference/ENAA/*.pdf')):
    base = os.path.basename(f)
    if base in EXPECTED: continue
    try:
        r = PdfReader(f)
        txt = (r.pages[0].extract_text() or '')[:500] if r.pages else ''
        m = re.search(r'G25-029-([0-9])', txt)
        results[base] = m.group(1) if m else 'UNKNOWN'
    except Exception:
        results[base] = 'ERR'
print(json.dumps(results, ensure_ascii=False))
"
```

注意：

- カレントディレクトリがリポジトリのルートであることを前提にしたパス。それ以外なら glob のパスを調整する
- `PYTHONIOENCODING=utf-8` は Windows 既定の cp932 で ENAA Copyright 記号が出力エラーになるのを回避するため必須
- Excel（`.xlsx`）の自動識別は実装が複雑になるため、`reference/ENAA/` 直下に **期待名以外の `.xlsx` が 1 個だけ** あれば G25-029-2 と推定してよい。複数あれば利用者に確認

### Step 4: リネーム提案（必ず確認を取ってから実行）

識別結果を以下のように整形して利用者に提示し、リネームしてよいか **1 問で確認** する：

```
ENAA 資料が異なる名前で配置されているようです。中身を確認したところ、以下のように識別できました：

  ダウンロード.pdf      → mes_g25-029-1.pdf  （解説資料）
  ダウンロード (1).pdf  → mes_g25-029-3.pdf  （クイックスタートガイド）
  ダウンロード (2).pdf  → mes_g25-029-4.pdf  （これまでの経緯）
  ダウンロード (3).pdf  → mes_g25-029-5.pdf  （How to Use）

この対応でリネームしてよろしいですか？（はい / いいえ）
```

利用者が「はい」と答えたら `mv`（PowerShell では `Move-Item`）で実行する。
リネーム後にもう一度 Step 1 を走らせて確認結果を報告する。

### Step 5: 識別不能な場合（`G25-029-X` パターンが見つからない）

- 「このファイルは ENAA 資料ではない可能性があります。中身を確認して、別物なら別フォルダへ移動してください」と案内
- ENAA 資料のはずなら、利用者に手動で番号を確認してもらう

### Step 6: ファイル自体がない場合（Step 3 で識別対象なし）

`reference/ENAA/` 直下に PDF も Excel も 1 つもなければ：

1. すぐに対話を始めず、`reference/ENAA/README.md` の取得手順を案内
2. ユーザーが「資料なしで始めたい」と明示した場合のみ、ENAA 12 大分類は仮の汎用カテゴリで進める旨を断った上で開始
3. その場合、出力には **「ENAA 標準準拠ではない、暫定版」と明記** して、後で ENAA 配置後に再生成することを促す
