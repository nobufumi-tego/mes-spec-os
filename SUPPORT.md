# サポート方針

本プロジェクトは **個人運営のオープンソースプロジェクト** です。
原則として **「ドキュメント参照 → 自己解決」** を基本としており、個別対応は最小限です。

## サポート区分

### Tier 0：自己解決（推奨・大半はこちらで解決可能）

以下を順に確認することで、ほとんどの疑問は解決します：

1. **`README.md` と `docs/` 配下のドキュメント**を読む
   - 特に `docs/セキュリティ.md` は利用前に必読
2. **ENAA 公式資料**を `reference/ENAA/` に配置済みなら直接読む
3. **Claude にエラーメッセージを貼り付けて相談**する
   - 「これ何？どうしたら直る？」の一文で十分
4. **過去の GitHub Issues を検索**する（公開後）

このスターターは「Claude と利用者だけで対話完結する」設計です。
本リポジトリのオーナーに連絡しなくても解決できる作りを目指しています。

### Tier 1：GitHub Issues（best-effort、SLA なし）

- バグ報告・機能要望は **GitHub Issues** へ（公開後に有効化）
- 対応は **best-effort**。返信時期・採否は保証しません
- 重複・既知の問題は告知なくクローズされる場合があります
- 機密情報を含む内容は Issue に書かないでください
- 個別の **商用利用相談・導入支援依頼** は Issues でも受け付けません

### Tier 2：個別対応（原則お受けしません）

以下の **限られた場合のみ** 例外的に対応します：

- **セキュリティ脆弱性の報告**：GitHub Security Advisories 経由で
- **ライセンス違反の指摘**：法的観点での連絡
- **ENAA 等の権利者からの正式連絡**：本人確認後に対応

それ以外の個別対応はすべてお受けしません。

## 対応しないこと（明示）

- ❌ 電話での技術サポート
- ❌ ZOOM / Teams 等のオンラインミーティング
- ❌ 対面でのコンサルティング・トレーニング
- ❌ メールでの個別技術質問への回答
- ❌ 顧客固有の業務適用支援（カスタマイズ請負）
- ❌ 24/7 のレスポンス保証
- ❌ SLA・保守契約

これらが必要な業務利用の場合、**有償の MES コンサルタント・SIer をご検討ください**。本スターターは「各社の検討の出発点となる雛形」を提供することが目的で、導入支援サービスではありません。

## セキュリティ問題の報告

技術的脆弱性（情報漏洩経路の発見・認証バイパス等）を発見した場合：

1. **公開 Issue には書かない**（攻撃者に情報を与えるため）
2. **GitHub Security Advisories** から非公開で報告（公開後）
3. 90 日以内に対応する努力をしますが、保証はしません

利用中に **自分の機密情報が漏洩した可能性に気づいた** 場合：

- 本リポジトリのオーナーへの報告は不要です
- `docs/セキュリティ.md`「7. リスクシナリオと対応」に従って自社内で対応してください
- 必要に応じて Anthropic（API 提供元）へ直接ご連絡ください

## 商用利用について

ENAA 資料の利用条件（自社内・RFP への二次利用可）の範囲内であれば、
本スターター自体は OSS ライセンス（MIT 等）の範囲で自由に商用利用できます。

ただし：

- **本リポジトリのオーナーは、商用利用に関する個別質問にお答えしません**
- 法的判断は自社の法務部門にご確認ください
- ENAA 資料の利用条件解釈に疑義がある場合は ENAA へ直接お問い合わせください

## このサポート方針の根拠

オープンソースプロジェクトでは、**メンテナの時間が最も希少なリソース** です。
個別対応に時間を使うほど、コア機能の改善・ドキュメント整備が遅れ、
結果として **すべての利用者が損をする** ことになります。

本プロジェクトは少人数（個人 / ごく小さなチーム）で運営することを前提としており、
「ドキュメントを読めば自己解決できる」状態を作ることに注力します。
個別対応をお断りすることで、より多くの方に長く価値を提供することを目指しています。

ご理解いただけますと幸いです。
